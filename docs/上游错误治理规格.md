# 上游错误治理规格

## 背景

当前网关已经具备上游错误透传规则、账号 failover、同账号短重试、运维错误日志等能力，但客户端收到的上游错误仍偏通用，管理员排查时也缺少稳定分类。第一期目标是在不破坏现有协议响应结构的前提下，对 OpenAI 上游错误做统一分类、友好提示和可观测性补强。

## 目标

- 客户端收到稳定、可读、不会泄露上游密钥或内部连接信息的错误提示。
- OpenAI Responses、Chat Completions、Anthropic Messages 兼容路径保持原有响应外形。
- 运维日志记录稳定的上游错误分类，便于搜索、告警和后续统计。
- 保留现有错误透传规则、账号 failover 与同账号重试行为。

## 范围

第一期覆盖：

- OpenAI 上游 HTTP 错误响应分类。
- OpenAI transport/network 错误最终响应体友好化。
- `OpenAIGatewayService.handleErrorResponse`。
- `OpenAIGatewayService.handleCompatErrorResponse`，即 Chat Completions 与 Anthropic Messages 兼容路径。
- `ops_error_logs.upstream_errors` JSON 中增加分类摘要，不改数据库 schema。

第一期不覆盖：

- Gemini、Antigravity、Claude 原生通道的完整分类改造。
- 已经由错误透传规则命中的响应内容。
- 流式响应已经向客户端写出数据后的错误重写。
- 新增自动请求重放策略。

## 分类规则

分类优先看上游 HTTP 状态码，再结合 `error.code`、`error.type`、`error.message`、顶层 `detail`/`message` 的关键词。

| 分类 | 典型信号 | 客户端状态 | 错误类型 | 客户端提示 |
| --- | --- | --- | --- | --- |
| `authentication_failed` | 401、`invalid_api_key`、`incorrect api key` | 502 | `upstream_authentication_error` | Upstream authentication failed, please contact administrator |
| `billing_required` | 402、`billing`、`insufficient_quota`、`quota exceeded` | 502 | `upstream_billing_error` | Upstream billing or quota issue, please contact administrator |
| `permission_denied` | 403、`forbidden`、`permission` | 502 | `upstream_permission_error` | Upstream access forbidden, please contact administrator |
| `rate_limited` | 429、`rate limit`、`too many requests` | 429 | `rate_limit_error` | Upstream rate limit exceeded, please retry later |
| `invalid_model` | 400，且包含 `model`、`not found`、`does not exist` | 400 | `invalid_request_error` | Requested model is not available |
| `model_unavailable` | 404，且包含 `model`、`not found`、`does not exist` | 502 | `upstream_model_error` | Upstream model is unavailable, please contact administrator |
| `context_length_exceeded` | `context length`、`maximum context`、`too many tokens` | 400 | `context_length_exceeded` | Request exceeds upstream context length limit |
| `content_policy` | `content policy`、`safety`、`policy_violation` | 400 | `content_policy_error` | Request was rejected by upstream content policy |
| `upstream_timeout` | 408、504、`timeout` | 502 | `upstream_timeout_error` | Upstream request timed out, please retry later |
| `upstream_server_error` | 500、502、503、529 | 502 或 503 | `upstream_server_error` | Upstream service temporarily unavailable, please retry later |
| `transport_error` | DNS、TCP、TLS、proxy 等没有 HTTP 响应的错误 | 502 | `upstream_transport_error` | Upstream connection failed, please retry later |
| `upstream_bad_request` | 400/422 且未命中更具体分类 | 502 | `upstream_error` | Upstream rejected the request |
| `unknown` | 未命中以上规则 | 502 | `upstream_error` | Upstream request failed |

## 响应兼容性

- Responses 非兼容路径继续返回 OpenAI 风格 JSON：

```json
{"error":{"type":"...","message":"..."}}
```

- Chat Completions 兼容路径继续返回 OpenAI Chat Completions 错误外形：

```json
{"error":{"type":"...","message":"..."}}
```

- Anthropic Messages 兼容路径继续通过既有 writer 返回 Anthropic 错误外形：

```json
{"type":"error","error":{"type":"...","message":"..."}}
```

- 已配置的错误透传规则优先级高于分类器。
- 自定义错误码未包含的状态码继续返回通用网关错误，避免暴露上游细节，但仍记录分类。

## 重试策略

第一期不新增自动重试策略，只复用现有能力：

- Service 层判断账号是否需要 failover。
- Handler 层在未写出流内容前执行账号切换。
- `RetryableOnSameAccount` 仍由现有账号池策略决定。
- 已经写出流内容后不再重写错误，也不自动重试。

后续第二期可基于分类扩展受控重试：

- 仅对 `rate_limited`、`upstream_timeout`、`upstream_server_error`、`transport_error` 等临时错误启用。
- 尊重 `Retry-After`。
- 仅在请求尚未向客户端写出内容时执行。
- 非幂等或大请求需要有明确保护。

## 可观测性

`ops_error_logs.upstream_errors` 中每个事件增加 `classification` 字段，记录稳定分类名，例如：

```json
{"classification":"rate_limited"}
```

该字段只包含稳定分类，不包含密钥、完整请求体、代理地址凭据或内部敏感链接。`detail` 继续用于保存已脱敏的上游错误摘要，避免影响错误透传规则的 `skip_monitoring` 匹配。

## 管理策略

后台“上游错误策略”默认列出全部内置分类。每个分类有默认返回状态码、错误类型、错误提示和默认是否可重试；未保存覆盖项时完全使用代码默认值。

管理员可对单个分类覆盖：

- 是否启用自定义返回。
- 返回 HTTP 状态码。
- `error.type`。
- `error.message`。
- 是否启用重试。
- 最大重试次数。
- 备注。

重试第一期复用现有 failover 机制：

- 仅在 service 层尚未写响应时触发。
- 开启重试后返回 `UpstreamFailoverError`，交给 handler 执行现有同账号重试/切换账号流程。
- 流式响应已经开始写出时不新增自动重试。
- 默认只建议临时错误分类启用重试：`rate_limited`、`upstream_timeout`、`upstream_server_error`、`transport_error`。

旧“错误透传规则”保留为同一入口下的“高级规则”，用于关键词、状态码、平台等特殊匹配场景；高级规则命中后仍优先于分类策略。

## 验收标准

- 401/402/403/429/5xx 等常见上游错误返回更准确的客户端提示。
- 上游返回上下文超限、模型不可用、内容策略拦截时可以被分类。
- Chat Completions 与 Anthropic Messages 兼容路径响应结构不变。
- 错误透传规则仍优先于分类器。
- 自定义错误码未处理的状态仍不暴露上游原始细节。
- 运维事件包含分类摘要。
- 后台可列出所有内置分类策略，并保存单个分类的返回提示和重试覆盖项。
- 相关单元测试覆盖分类器、OpenAI 非兼容路径和兼容路径。
